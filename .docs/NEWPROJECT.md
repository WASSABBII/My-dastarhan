# RestoBook v2 — Платформа бронирования с интерактивным планом зала

> Версия: 0.4
> Дата: 2026-04-02
> Рынок: Казахстан (приоритет), СНГ (перспектива)

---

## Концепция продукта

**RestoBook** — SaaS-платформа для ресторанов Казахстана.

Клиент заходит на страницу ресторана, видит **интерактивный план зала**, выбирает свободный столик (у окна, в углу, на террасе), указывает дату/время/гостей и бронирует. За час до визита WhatsApp или Telegram бот отправляет напоминание и ждёт подтверждения.

Аналог: **выбор мест в кинотеатре**, только для ресторана.

---

## Что изменилось по сравнению с v1

| v1 (старая концепция) | v2 (новая концепция) |
|----------------------|---------------------|
| AI-агент создаёт брони в WhatsApp | Брони через сайт, интерактивный план зала |
| Claude Haiku для понимания текста | Простая rule-based логика для броней |
| Клиент пишет боту — бот бронирует | Клиент выбирает столик как в кинотеатре |
| WhatsApp = основной канал брони | WhatsApp/Telegram = только уведомления |

---

## Роли в системе

| Роль | Кто | Доступ |
|------|-----|--------|
| `super_admin` | Команда RestoBook | Вся платформа, одобрение ресторанов |
| `owner` | Владелец ресторана | Полный доступ к своему ресторану/сети |
| `staff` | Сотрудник ресторана | Только управление бронями |
| `client` | Гость ресторана | Каталог, бронирование, отзывы |

## Флоу: Клиент

### Регистрация и вход
```
1. Вводит номер телефона (+7 777 123 45 67)
2. Получает OTP-код в WhatsApp или SMS
3. Вводит код → авторизован
4. Заполняет имя (один раз)
```

### Каталог ресторанов
```
restobook.kz/catalog

- Список ресторанов с фото, рейтингом, кухней, районом
- Поиск по названию
- Фильтры: район, кухня, вместимость, рейтинг
```

### Страница ресторана
```
restobook.kz/{slug}  →  например restobook.kz/baraka

Вкладки:
├── О ресторане  — описание, фото галерея, короткое видео, адрес, часы
├── Меню         — категории, блюда, цены, фото блюд, аллергены
├── Бронирование — интерактивный план зала
├── Отзывы       — своя система + Google оценка
└── AI-чат       — задать вопрос про ресторан
```

### Бронирование
```
1. Выбирает дату и время
2. Выбирает примерную длительность: 1 час / 1.5 часа / 2 часа / 3 часа
3. Видит план зала:

   ┌──────────────────────────────────────────┐
   │  🪟 Окно                                  │
   │  [T1✓] [T2✓] [T3✗]   ✓ свободен         │
   │                        ✗ занят            │
   │        [T4✓] [T5✗]                        │
   │                                            │
   │  🌿 Терраса                                │
   │  [T6✓] [T7✓]                              │
   │                        🚪 Вход            │
   │        [T8✗] [T9✓]                        │
   │  🔒 VIP                                    │
   └──────────────────────────────────────────┘

4. Кликает на свободный столик
5. Подтверждает бронь (данные уже из профиля)
6. Получает подтверждение в WhatsApp / Telegram
```

### Личный кабинет клиента
```
- История броней (предстоящие + прошлые)
- Отмена предстоящей брони
- Оставить отзыв после визита
- Настройки профиля (имя, канал уведомлений)
```

## Флоу: Ресторан / Админ

### Регистрация
```
1. Заходит на restobook.kz/for-restaurants
2. Заполняет форму: название, адрес, контакты, email
3. Статус → "на рассмотрении"
4. SuperAdmin одобряет → владелец получает письмо
5. Выбирает тариф и оплачивает (или пробный период 30 дней)
6. Попадает в админ-панель → начинает настройку
```

### Настройка ресторана (один раз)
```
Шаг 1 — Основная информация:
- Название, описание, кухня, адрес, телефон
- Часы работы по дням недели
- Загрузка фото ресторана (до 10 фото, JPG/PNG, макс 5 МБ каждое)
- Загрузка короткого видео (1 видео, MP4, макс 50 МБ, до 60 сек)

Шаг 2 — Меню:
- Создать категории (Горячее / Напитки / Десерты и т.д.)
- Добавить блюда: название, описание, цена, фото (JPG/PNG, макс 2 МБ), аллергены

Шаг 3 — План зала:
- Добавить столики: метка (T1, VIP-1), вместимость, зона, координаты на плане

Шаг 4 — Уведомления:
- Указать WhatsApp номер для алертов менеджеру
- Подключить Telegram (если нужно)
```

### Ежедневная работа
```
Панель администратора:
- Канбан броней: Ожидает → Подтверждено → Пришли → Не пришли
- Ручная отмена любой брони (с уведомлением клиента)
- Ручное бронирование: выбор нескольких столиков сразу (корпоратив, VIP)
- Realtime: новые брони появляются мгновенно
- Статус "Задержался": хостес нажимает кнопку → следующий гость получает уведомление
- Одобрение/отклонение запросов на продление от гостей
- Статистика: загруженность, no-show rate, популярные столики
- Скачать QR-коды PDF для столиков
- Управление базой знаний AI-чата (FAQ, акции)
- Просмотр отзывов и ответы на них
```

### Несколько аккаунтов на один ресторан
```
Owner может добавить staff:
- Вводит email сотрудника → тот получает приглашение
- Staff видит только брони (без настроек, без финансов)
- Owner может удалить доступ в любой момент
```

### Сеть ресторанов
```
Владелец нескольких точек:
- Один аккаунт → дропдаун для переключения между точками
- Общая аналитика по всей сети
- Меню можно синхронизировать или делать отдельным на каждую точку
```

## WhatsApp / Telegram — сценарии уведомлений

### Сценарий 1: Подтверждение брони
```
Сразу после брони:
─────────────────────────────────────────
Ресторан "Барака" подтверждает вашу бронь:
📅 Пятница, 4 апреля в 19:00
🪑 Столик T1 (у окна), 2 гостя
📍 ул. Абая 10, Алматы

Отменить бронь: restobook.kz/cancel/TOKEN
─────────────────────────────────────────
```

### Сценарий 2: Напоминание за 1 час
```
─────────────────────────────────────────
Привет! Напоминаем:
Ваш столик T1 (у окна) в ресторане "Барака"
сегодня в 19:00 на 2 человек.

Вы придёте?
Ответьте ДА или НЕТ
─────────────────────────────────────────

ДА       → booking.status = "confirmed"
НЕТ      → предупреждение об отмене, ждём ОТМЕНА
ОТМЕНА   → booking.status = "cancelled", столик свободен
(тишина 30 мин) → operator_alert → уведомление менеджеру
```

### Сценарий 3: Запрос отзыва после визита
```
Через 2 часа после статуса "пришли":
─────────────────────────────────────────
Спасибо за визит в "Барака"!
Оцените ваш опыт: restobook.kz/review/TOKEN
─────────────────────────────────────────
```

### Сценарий 4: Уведомление менеджеру (нет ответа)
```
─────────────────────────────────────────
⚠️ Клиент не подтвердил бронь
T1 (у окна) в 19:00 — 2 гостя
Телефон: +7 777 123 4567
Столик ждать или освободить?
─────────────────────────────────────────
```

### Сценарий 5: Уведомление за 15 мин до конца брони
```
BullMQ job срабатывает за 15 мин до (time_start + estimated_duration):
─────────────────────────────────────────
Ваше время в "Барака" заканчивается через 15 минут (в 20:00).
Всё ещё у нас?

[Остаюсь — места хватает]
[Нас стало больше — нужен стол побольше]
[Скоро уходим]
─────────────────────────────────────────

[Остаюсь, места хватает]:
  → Следующая бронь есть → запрос на продление хостесу
  → Следующей брони нет  → автопродление, гость сидит

[Нас стало больше]:
  → Бот: "Сколько вас теперь?"
  → Гость пишет число (например: 5)
  → Система ищет свободный стол нужного размера:
      Есть свободный → автобронь (данные из текущей брони) + WhatsApp подтверждение
      Нет свободного → "Ближайшие столики освободятся:
                        - Стол №2 — через ~20 мин
                        - Стол №5 — через ~35 мин
                        Хотите встать в очередь? [Да, жду стол №2]"
  → Гость жмёт "Да" → запись в waitlist → уведомление когда освободится

[Скоро уходим]:
  → Просто фиксируем, ничего не делаем

Если нет свободных столиков (изначально):
─────────────────────────────────────────
Свободных столиков сейчас нет.
Ближайшие освободятся:
- Стол №2 (4 чел) — через ~20 мин
- Стол №5 (2 чел) — через ~35 мин

Хотите занять место в очереди? [Жду стол №2]
─────────────────────────────────────────
```

### Сценарий 6: Уведомление следующему гостю (стол задержался)
```
Хостес нажимает "Задержался" в панели:
─────────────────────────────────────────
Добрый вечер! Ваш столик в "Барака" немного задерживается.
Ориентировочно освободится через 15–20 минут.
Приносим извинения, угощаем вас напитком в ожидании 🙏
─────────────────────────────────────────
```

## AI-чат для клиентов

```
┌─────────────────────────────────────┐
│  💬 Спросите про ресторан           │
├─────────────────────────────────────┤
│  Бот: Привет! Спрашивай про "Барака"│
│                                     │
│  Клиент: Есть ли детское меню?      │
│                                     │
│  Бот: Да — пицца, паста, нагетсы.  │
│  Также есть высокие стульчики 🪑    │
│                                     │
│  Хотите забронировать? [Выбрать стол]│
└─────────────────────────────────────┘
```

**Как работает:**
```
1. Ресторан заполняет базу знаний: меню, FAQ, акции, контакты
2. Клиент пишет → POST /chat/{slug}
3. Backend: system prompt = инфо ресторана, user = вопрос клиента
4. Claude Haiku отвечает за ~1 сек (~$0.001 за вопрос)
5. История чата — в сессии браузера (не в БД)
```

**Умеет:** отвечать на вопросы по меню/ценам/условиям, помогать выбрать столик, переключать на бронирование, отвечать на русском/казахском/английском.

**Не умеет:** бронировать самостоятельно.

## Отзывы

**1. Google Places (внешние)**
- Подтягиваем по `place_id` ресторана
- Показываем рейтинг + последние 5 отзывов
- Кэш в Redis, обновление раз в сутки
- Бесплатно до 5000 запросов/мес

**2. Своя система (после визита)**
- Триггер: статус брони сменился на "пришли" → через 2 часа WhatsApp/Telegram
- Клиент ставит 1–5 + текст (опционально)
- Ресторан видит отзывы и может ответить в админке

## Поток данных: бронирование (с защитой от гонки)

```
1. GET /restaurants/baraka/availability?date=2026-04-04&time=19:00
   → список table_id со статусом (free / busy)
   → учитывает buffer_minutes ресторана при расчёте

2. Клиент кликает T1 → открывается форма
   → выбирает estimated_duration (1ч / 1.5ч / 2ч / 3ч)

3. POST /bookings
   → BEGIN TRANSACTION
   → time_end = time_start + estimated_duration + buffer_minutes
   → SELECT * FROM bookings
        JOIN booking_tables bt ON bt.booking_id = bookings.id
        WHERE bt.table_id = T1
        AND date = '2026-04-04'
        AND time_start < time_end AND time_end > time_start
        AND status NOT IN ('cancelled')
        FOR UPDATE                    ← блокировка строки
   → если занято → вернуть 409 Conflict
   → если свободно → INSERT booking + INSERT booking_tables
   → COMMIT
   → BullMQ: задача "send_reminder" через (time_start - 1ч)
   → BullMQ: задача "end_reminder" через (time_start + estimated_duration - 15мин)
   → Twilio/Telegram: подтверждение клиенту

4. Socket.io broadcast → все видят T1 как занятый

5. За 1 час → BullMQ → напоминание клиенту (ДА/НЕТ/ОТМЕНА)

6. За 15 мин до конца → BullMQ → "end_reminder":
   → Проверяем свободные столики в зале
   → Отправляем уведомление с кнопками (Сценарий 5)

7. Webhook от клиента → обработка ответа:
   "Остаюсь"         → проверить следующие брони → продлить или запрос хостесу
   "Нас стало больше" → бот запрашивает количество → поиск стола → автобронь или waitlist
   "Скоро уходим"    → фиксируем, ничего не делаем
   "Жду стол №X"     → INSERT waitlist → уведомление при освобождении

8. Ручная бронь через админку:
   → created_by = 'staff'
   → Мультиселект столиков на флор-плане
   → INSERT booking + несколько записей в booking_tables
```

## Дополнительные фичи MVP

### QR-код на столике
```
1. Ресторан нажимает "Скачать QR-коды" в админке
2. pdfkit генерирует PDF — один QR на каждый столик
3. QR ведёт на restobook.kz/baraka?table=T1
4. Клиент сканирует → страница ресторана, столик T1 подсвечен
```

### Онлайн-меню
```
Вкладка "Меню" на странице ресторана:
- Категории + блюда + цены + фото
- AI-чат автоматически использует меню как контекст
```

### Waitlist (лист ожидания)
```
Все столики заняты → клиент оставляет телефон
Кто-то отменяет → BullMQ триггер → WhatsApp/Telegram:
"Столик освободился! Забронируйте за 15 минут: [ссылка с токеном]"
```

### Telegram бот
```
Клиент выбирает канал при регистрации: WhatsApp / Telegram
Та же логика: подтверждение → напоминание → ДА/НЕТ → алерт менеджеру
Telegram бесплатный → экономия на Twilio
```

| WhatsApp уведомления | Низкая | BullMQ delayed job + Twilio |
| Telegram уведомления | Низкая | Telegraf.js |
| Обработка ответов (Да/Нет) | Низкая | Webhook + switch/case |
| OTP верификация клиента | Низкая | Twilio SMS + Redis TTL |
| Канбан-доска для ресторана | Средняя | @dnd-kit + WebSocket |
| Настройка плана зала | Низкая | Форма с координатами столиков |
| Загрузка фото и видео | Низкая | react-dropzone + Cloudinary API |
| Статистика | Низкая | SQL агрегации + Recharts |
| Аутентификация (users) | Низкая | JWT + bcrypt |
| Управление сотрудниками | Низкая | Invite по email + restaurant_users |
| Подписки и тарифы | Средняя | subscriptions таблица + feature gates |
| SuperAdmin панель | Низкая | CRUD ресторанов + статусы |
| AI-чат | Низкая | Claude Haiku + база знаний |
| Отзывы — своя система | Низкая | Токен + форма оценки |
| Отзывы — Google Places | Низкая | API запрос + кэш Redis |
| QR-коды + PDF | Низкая | qrcode + pdfkit |
| Онлайн-меню | Низкая | CRUD + фото + вкладка на странице |
| Waitlist | Низкая | BullMQ триггер при отмене |
| Сеть ресторанов | Низкая | restaurant_groups + дропдаун |
| Уведомление за 15 мин до конца брони | Низкая | BullMQ end_reminder + выбор свободных столиков |
| Автопродление / запрос хостесу | Низкая | Webhook + проверка следующих броней |
| Пересадка на другой стол (автобронь) | Средняя | Поиск свободного → INSERT booking из данных текущей брони |
